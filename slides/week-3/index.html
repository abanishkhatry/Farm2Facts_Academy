Building the shared version.

The merged CLAUDE.md takes the best of all three. Here is what goes in:

Simulation step call graph, dead code table, preprocessing docs. The technical map an agent needs to navigate the codebase.

From B

Process + guardrails

Current-state caveats, "How to Work With Me" section, spec-before-code guidance. The rules an agent needs to follow.

From C

Operations + standards

Deployment config levers, logging standards, agentic workflow rules, architecture constraints. The conventions that prevent mistakes.

What stays individual?

Platform notes (Windows vs Mac), personal agent preferences, and workflow style belong in your personal context file, not the shared one. The shared file captures what every contributor and every agent session needs to know.

Layer 3

LLM adversarial review.

Open a fresh session (no prior context). Paste the diff with a skeptical reviewer prompt.

Prompt
Review this diff for correctness, performance, and maintainability issues.
Be thorough and skeptical. For each finding, rate your confidence
and explain why it matters.
  • 1
    Read every finding
    The tool returns a mix of real issues and false positives.
  • 2
    Post as PR comments
    Mark each: "Valid" or "False positive (because...)"
  • 3
    Learn the ratio
    Filtering signal from noise is the core skill here.
The rotation: Each PR gets a designated adversarial reviewer (not the author, not the peer reviewer). Rotation assigned at the start of each week.

The pattern

Write before you code.

Before implementing, write a short spec describing what you will change, why, and how you will verify it worked.

design decisions

Write an ADR

Choosing between approaches? Changing behavior? Use the ADR template: context, decision, alternatives.

straightforward fixes

Write a diagnosis paragraph

Bug fix? Simple cleanup? A paragraph in the issue comment is enough: what is broken, what causes it, how you will fix it.

This week: #7 (optimization) writes an ADR. #6 (bug fix) writes a diagnosis. #8 (redundancy) maps the codebase first. #9 (logging) writes a config spec.
Beyond this project: The spec-first habit separates junior from senior professionals in any field. Writing down your plan before executing protects you from building the wrong thing and creates a record of reasoning for future readers.

Week 3 · Specs before code • Planning tools
Planning tools

What each is good at.

Your hand-written spec

  • Captures intent and constraints
  • Explains why this approach
  • Identifies what not to do
  • Considers team and project context

Agent-generated plan

  • Maps file-level specifics
  • Identifies dependencies you missed
  • Suggests test strategies
  • Catches call sites across the codebase

Best workflow:

Write your spec first (the what and why), then feed it to /plan as context to get the how.

Structured planning is most useful for single-feature implementations. It is not a substitute for deciding whether to do something. That is the ADR.
